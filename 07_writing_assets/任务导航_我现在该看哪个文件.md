# 任务导航：我现在该看哪个文件

这个文件按“你现在要做什么”来导航。不要按文件顺序读，按任务读。

## 1. 我要读懂一篇论文

看：
- `论文阅读卡片模板.md`
- `从一篇论文拆出写法.md`

你要产出：
- 这篇论文解决什么问题。
- 它的方法一句话怎么说。
- 它最值得学的图表/句式是什么。
- 它的 limitation 是什么。

## 2. 我要找 gap

看：
- `related_work_matrix.csv`
- `如何看懂related_work_matrix.md`

操作：
- 先筛 topic。
- 再看 limitation。
- 把多个 limitation 合并成一个你能实验验证的 gap。

例子：
如果很多 FPGA accelerator 都说 memory bandwidth 或 resource overhead 是瓶颈，你的 gap 就不能写成“现有方法不好”，而要写成“现有方法在某资源约束下缺少对 memory/dataflow 的联合优化”。

## 3. 我要写摘要

看：
- `abstract_intro_sentence_bank.md`
- `paper_skeleton_templates.md`

操作：
- 先选 brief、regular paper、EDA/tool、安全硬件或 FPGA accelerator 型。
- 再填：问题、方法、平台、指标、结论。

## 4. 我要写 Introduction

看：
- `abstract_intro_sentence_bank.md`
- `从自己结果到论文初稿.md`

操作：
- 第一段写应用背景。
- 第二段写硬件瓶颈。
- 第三段写现有方法不够。
- 第四段写你的贡献。

## 5. 我要做 related work

看：
- `related_work_matrix.csv`
- `如何看懂related_work_matrix.md`

操作：
- 选 8-12 篇最相近论文。
- 按方法类型分组，不按年份流水账。
- 每组写：别人做了什么、还缺什么、你的方法补哪里。

## 6. 我要做图表

看：
- `figure_table_pattern_library.md`

操作：
- 第一张图通常是 system overview 或 architecture figure。
- 方法部分用 dataflow / pipeline / memory hierarchy 图。
- 实验部分必须有 SOTA comparison table。

## 7. 我要选 venue

看：
- `venue_pattern_matrix.csv`

快速判断：
- 单点创新、篇幅短：TCAS-II。
- 完整电路/系统闭环：TCAS-I。
- VLSI 架构和实现效率：TVLSI。
- 系统/体系结构意义强：TC。
- EDA/tool/design flow：TCAD。
- 密码硬件/安全评估：TCHES/CHES。

## 8. 我要投稿前自查

看：
- `reviewer_checklist.md`

重点查：
- claim 有没有证据。
- baseline 是否公平。
- SOTA 表是否归一化。
- 是否说明限制条件。
